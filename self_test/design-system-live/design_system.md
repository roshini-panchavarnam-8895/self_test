
# Enterprise Design System Documentation

This document defines the **Design System Library** for this repository, covering tokens, icons, components, form fields, report layouts, documentation standards, and governance rules.

The system is centered on three ideas:

- **Reusable enterprise form field components** under `forms/`
- **Report layout patterns** under `reports/`
- **Token-driven implementation with Figma ↔ code sync** through `figmasync/`

# 1. Home

## Overview

This project is an **enterprise design system workspace** used to document, preview, and maintain a library of form field and report UI patterns.

It provides:

- Design tokens expressed as **CSS custom properties** and SCSS variables
- A generated **design library** (`design-library.html`) built from source components
- Standardized **form field components** with a strict `zc-` class naming convention
- **Report layout patterns** including calendar, kanban, timeline, list, and card-style views
- **Smooth icon fonts** for outline and solid visual affordances
- A **two-way sync layer** between Figma variables and code tokens via `figmasync/auto-sync.mjs`

## Core Implementation Notes

- The design library UI is generated by `_build-design-library.js`
- Form components live in separate folders under `forms/`
- Report patterns live under `reports/`
- Token synchronization is mapped in `figmasync/design-map.json`

# 2. Design Tokens

Design tokens are the **visual foundation of the design system**. In this project, tokens are implemented as CSS variables, supported by SCSS variables, and selectively synced with Figma variables.

Primary sources include:

- `color_variables.css`
- `_general_variables.scss`
- `figmasync/design-map.json`

## Typography

The form field system uses a compact enterprise scale designed for dense data-entry layouts.

| Token | Value | Usage |
|------|------|------|
| `--font-family` | `'Lato', sans-serif` | Base UI font for form controls and labels |
| `--font-size-base` | `14px` | Labels, input text, checkbox/radio text |
| `--font-size-sub-label` | `13px` | Placeholder text, sub-labels, helper text |
| `--font-size-section` | `16px` | Section separators and internal section titles |
| `$zc-font-regular` | `14.5px` equivalent | SCSS typography scale used in legacy/shared styling |

Example:

```html
<label class="form-label">
  <span class="zc-label-text">Single Line</span>
</label>
```

## Grid System

The repository includes a **12-column utility system** in `grid.css` for layout-level composition, especially for larger pages and report containers.

- Column classes follow the pattern `zc-col-1` to `zc-col-12`
- Responsive variants are available such as `zc-col-lg-*` and `zc-col-md-*`
- These utilities are suitable for page and report layout shells

```css
.zc-col-6 {
  flex: 0 0 calc(50% - var(--column-gap) * 0.5);
  max-width: calc(50% - var(--column-gap) * 0.5);
}
```

---

## Flex Utilities

Shared page- and report-level layouts may use flex-based utilities and alignment patterns. However, **form field component internals must not be converted to flex or grid** unless the design system rules are explicitly updated.

Current rule of thumb:

- **Allowed:** page shells, report headers, report card wrappers, grid utilities
- **Not allowed for form field internals:** replacing field anatomy with flexbox/grid
- **Required for form field internals:** `display: table` / `table-cell` layout for label-input alignment

```css
.zc-report-row {
  display: flex;
  flex-wrap: wrap;
  gap: 16px;
}
```

## Common Structure

All form fields follow a consistent HTML structure using the standard class naming convention.

```text
form-group          ← Parent wrapper for every field
├── form-label      ← Label element
│   └── zc-label-text  ← Label text wrapper
│       └── fieldMandate  ← Mandatory indicator (*)
├── form-field      ← Input / control wrapper
│   ├── form-control       ← The actual input, dropdown, textarea, etc.
│   ├── zc-help-txt           ← Help / hint text
│   └── fieldErrorMsg   ← Error message (shown on validation failure)
```

### Validation Error State

Add the class `validationError` to `form-field` to activate the error state styling.

### Implementation Rules

1. **Table-based layout** — form field anatomy uses `display: table` / `table-cell` for label-input alignment
2. **Consistent class names** — use: `form-group`, `form-label`, `form-field`, `form-control`, `fieldMandate`, `fieldErrorMsg`, `zc-help-txt`
3. **Design tokens only** — avoid hardcoded color, typography, radius, or spacing values. Use `--zc-*` CSS custom properties
4. **SCSS architecture** — each component uses `@use '../common' as c;` to import shared styles from `forms/_common.scss`
5. **WCAG compliance** — wire labels, descriptions, and validation states correctly
6. **Code parity** — source shown in the design library must match the rendered component
7. **Separate files per component** — each component keeps its own `.html`, `.css`, `.scss`, and optional `.js`

---

## Example: Text Input

### Preview

Single-line text field with label, help text, and validation messaging.

### HTML

```html
<div class="form-group clearfix">
  <label class="form-label" for="zc-Single_Line">
    <span class="zc-label-text">Single Line <span class="fieldMandate" aria-hidden="true">*</span></span>
  </label>
  <div class="form-field clearfix validationError">
    <input class="form-control" id="zc-Single_Line" type="text" name="Single_Line"
           maxlength="255" aria-required="true"
           aria-describedby="zc-Single_Line-helpText zc-Single_Line-errorMsg"
           aria-invalid="true" autocomplete="off">
    <div id="zc-Single_Line-helpText" class="zc-help-txt">This is a help text</div>
    <div id="zc-Single_Line-errorMsg" class="fieldErrorMsg" role="alert">
      <span>This field is mandatory. Enter a value.</span>
    </div>
  </div>
</div>
```

### Accessibility Requirements

| Attribute | Where | Purpose |
|------|------|------|
| `for` / `id` | label → input | Associates label with control |
| `aria-required="true"` | input | Marks field as required |
| `aria-labelledby` | input | Points to the label id |
| `aria-describedby` | input | Connects help text and error text |
| `aria-invalid="true"` | input | Signals current validation failure |
| `role="alert"` | error container | Announces validation changes |
| `aria-live="assertive"` | error container | Immediate announcement for assistive tech |

# 8. Creating New Components or Fields

When a new UI element is required, teams must build it **from the existing project conventions first**, not from a blank pattern.

## Reuse First

Always check whether the requirement can be solved by extending an existing field or report pattern.

| Requirement | Base Component |
|------|------|
| Searchable option selection | Dropdown or Multi Select |
| Monetary entry | Currency Input |
| Boolean choice | Decision Box or Checkbox |
| Structured identity entry | Name Field |
| Structured location entry | Address Field |
| Media attachment | File/Image/Audio/Video Upload |

---

## Steps to Create a Component

1. Identify the closest existing folder under `forms/` or `reports/`
2. Reuse the established `zc-` naming and anatomy
3. Use design tokens instead of hardcoded values
4. Follow spacing, typography, and icon rules already used in the system
5. Define variants and states only if the use case requires them
6. Add accessibility wiring and keyboard behavior
7. Keep HTML, CSS, SCSS, and JS aligned with the final rendered output
8. Make sure the component can be surfaced correctly in the generated design library
9. If new synced tokens are introduced, update the sync map and documentation

All components must follow **WCAG 2.1 AA**.

- Visible focus states
- Screen-reader labels and relationships
- Color contrast compliance
- Error messaging that is announced properly
- Decorative icons hidden from assistive technology where appropriate

# 10. Design Token Governance

Tokens centralize design decisions and are part of the project’s **design-to-code synchronization workflow**.

## Token Rules

- No hardcoded visual values when a token already exists
- Always use CSS custom properties for component-level colors, type, sizing, and radius
- Prefer the synced token set for shared form-field decisions
- Keep token naming reusable and semantic
- When a token is part of Figma sync, update `figmasync/design-map.json` instead of creating drift between code and design

Example:

```css
background: var(--color-brand);
font-size: var(--font-size-base);
border-radius: var(--input-border-radius);
```

## Sync Governance

The sync workflow is handled by `figmasync/auto-sync.mjs`.

- Figma variables are read through the Desktop Bridge workflow
- CSS variables are mapped through `figmasync/design-map.json`
- Differences are reconciled between design tokens and code tokens
- The configured code-side token source is tracked in the sync metadata

# 11. Component Versioning

MAJOR.MINOR.PATCH

| Version | Meaning |
|------|------|
| Major | Breaking markup, token, behavior, or migration changes |
| Minor | New components, new variants, non-breaking enhancements |
| Patch | Fixes to styling, accessibility, docs, or implementation bugs |

# 12. Component Deprecation

Components may be deprecated when:

- A better standardized component already exists
- The implementation is no longer aligned with form-field rules
- Accessibility issues cannot be addressed safely in place
- The pattern duplicates another supported component
- The component no longer fits the token or sync strategy

Process:

1. Mark the component as deprecated in documentation
2. Provide a clear replacement component or pattern
3. Document migration guidance
4. Keep support through the agreed transition window
5. Remove in the next major release where appropriate

# 13. Migration Guidelines

When migrating older UI into this system, move toward **tokenized, prefixed, documented components**.

Example:

Old

```html
<button class="btn-old">Save</button>
```

New

```html
<button type="button" class="zc-btn zc-btn-primary">Save</button>
```

Additional migration expectations:

- Replace hardcoded colors with tokens
- Replace unprefixed classes with `zc-` classes
- Add ARIA relationships for labels, help text, and errors
- Preserve folder-based component organization
- Ensure the updated component can be represented correctly in the design library

- Follows `zc-` naming rules

- Keeps separate source files organized correctly
- Updates token sync mapping if new shared tokens were introduced
