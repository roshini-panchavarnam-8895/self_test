AI tools for your
daily work

A simple, practical guide to using AI tools wisely. No prior AI experience needed — just read, try, and learn.

Your first 5 minutes with AI

1 Open Claude Chat at claude.ai — it's the simplest starting point
2 Tell it what you're working on: "I'm a frontend dev working on a SCSS design system"
3 Ask your question with details: "How should I structure my color tokens for light and dark themes?"
4 If the answer isn't perfect — don't restart. Say "Make it simpler" or "Show me an example"
5 Always read and understand what AI gives you before using it
First things first

Know your tools

Think of these as different team members, each with a specialty. You wouldn't ask the same person for every task — same idea here.

C
Claude Chat
Think of it as: A smart colleague you can talk to
Ask questions, upload files, paste code, get explanations. Best for thinking, learning, and getting quick answers. Works at claude.ai in your browser.
CC
Claude Code
Think of it as: A developer who works in your terminal
Reads your files, runs commands, writes code across your project. Best for building things, scanning codebases, and automating tasks.
BE
Claude Browser Extension
Think of it as: An assistant looking over your shoulder
Works on any webpage. Highlight text or code on any site and ask questions about it — no copy-pasting needed.
CW
Claude Cowork
Think of it as: A project partner who creates files
Creates documents, pages, and polished outputs. Best for making things you'll share — reports, reference pages, documentation.
Cu
Cursor IDE
Think of it as: VS Code with a built-in AI pair programmer
A code editor that understands your entire project. Ask questions about your code, generate new files, and refactor — all with full project context.
GC
GitHub Copilot
Think of it as: Auto-complete on steroids
Lives inside VS Code and suggests code as you type. Also reviews pull requests on GitHub. Best as a always-on coding companion.
Understand the building blocks

Key concepts you'll hear about

These terms come up constantly in the AI tools world. You don't need to memorize them — just understand what they mean so you're not lost when someone mentions them.

A
Agent
Think of it as: A worker who can plan and act on their own
A regular AI just answers questions. An agent can actually do things — read your files, run commands, create folders, install packages, browse the web, and execute multi-step tasks. You give it a goal, and it figures out the steps to get there.

Claude Code is an agent — you say "set up a React project with TypeScript and add RTL support" and it creates folders, writes files, installs packages, and configures everything. Cursor Composer is also agentic — it can edit multiple files to accomplish a task.
How an agent works
You say: "Add dark mode to my app"
Reads your code → Plans the changes → Creates CSS variables → Updates components → Adds toggle button → Tests it
SA
Sub-agent
Think of it as: A helper the main agent delegates tasks to
When a task is complex, the main agent can spawn sub-agents — smaller, focused workers that handle specific parts of the job. The main agent coordinates everything, and each sub-agent handles one piece.

For example, in Claude Code, if you ask "refactor all components to use design tokens," the main agent might create sub-agents: one to scan existing hardcoded values, one to map them to tokens, and one to apply the changes — all working in parallel.
Main agent → Sub-agent: Scan files & Sub-agent: Map tokens & Sub-agent: Apply fixes
S
Skills
Think of it as: Instruction manuals you give to AI for specific tasks
A skill is a set of rules and instructions that teach AI how to do a specific type of task well. Instead of explaining your conventions every time, you write them once as a skill, and AI follows them automatically.

Skills are powerful because they make AI consistent. Without a skill, you'd need to repeat "use BEM naming with .zc- prefix" in every prompt. With a skill, AI knows this by default.
What a skill contains
A component library skill might include: which 28 components exist, what naming convention to use (BEM with .zc- prefix), what folder structure to follow, what grid system to use, which tokens are available, and how to handle RTL — all in one file that AI reads before starting work.
MD
SKILL.md & markdown config files
Think of it as: A cheat sheet you give AI before it starts working
Skills are written as markdown files (files ending in .md). The most common one is SKILL.md — a plain text file with structured instructions that AI tools can read and follow.

Different tools use different config files, but they all serve the same purpose — giving AI context about your project without you having to repeat it every time.
SKILL.md
Used by Claude Code and Claude Chat. Defines rules for a specific task type — like "how to build components for our design system."
CLAUDE.md
Project-level instructions for Claude Code. Lives in your repo root. Claude reads it automatically every session — your project's conventions, stack, and rules.
.cursorrules
Same idea but for Cursor IDE. Put your project conventions here and Cursor follows them in every suggestion and generation.
.github/copilot-instructions.md
GitHub Copilot's version. Sits in your repo and tells Copilot about your coding standards, naming patterns, and preferences.
Example: What goes in a CLAUDE.md or SKILL.md
# Project: Creator Design System ## Tech stack - SCSS with BEM naming (.zc- prefix) - HTML5 templates - RTL support required (use CSS logical properties) ## Conventions - Token variables: $zc-color-*, $zc-spacing-*, $zc-font-* - Component folder: /components/[name]/[name].scss + [name].html - Motion presets: snap, shift, pop, ease, enter, flourish, glide ## Rules - Never use hardcoded colors — always use $zc-color tokens - All spacing must use $zc-spacing tokens - Every component must support RTL via logical properties
M
MCP (Model Context Protocol)
Think of it as: Plug-ins that connect AI to external tools
MCP lets AI tools connect to other services — like Figma, Google Drive, Slack, Jira, or your internal tools. Instead of copying data between apps, AI can directly read from and write to these services.

For example, with MCP you could ask Claude to "read the latest design from Figma and generate matching component code" or "check my Jira tickets and summarize what's pending" — all in one conversation.
You ask Claude → MCP connector → Figma / Drive / Jira → AI processes data → Result to you
CW
Context window & Artifacts
Think of it as: AI's short-term memory and its scratchpad
The context window is how much text AI can "see" at once in a conversation — like a desk that can only hold so many papers. When a conversation gets very long, older parts may fall off the desk. That's why starting fresh with key context sometimes helps.

Artifacts are the outputs AI creates that you can preview, edit, and download — like HTML pages, React components, documents, or code files. In Claude Chat, artifacts appear in a side panel where you can see them live.
Why this matters in practice
If you're in a long conversation and AI starts forgetting earlier instructions, it's because the context window is full. Start a new chat and re-share the key context (or use a CLAUDE.md / SKILL.md file so AI picks it up automatically).
All these concepts connect: you write a SKILL.md file with your project rules → an agent (like Claude Code) reads it → the agent may use sub-agents for complex tasks → it connects to external tools via MCP → and produces artifacts you can use. Understanding this flow is understanding how modern AI-assisted development works.
Which brain to use

AI models — the quick guide

Each tool runs on an AI "model" — the brain behind it. Different models are better at different things. Here's what you need to know to pick the right one.

AI
What's a model?
Think of it as: The engine inside the car — same car, different engine = different performance
When you use Claude Chat, Cursor, or Copilot — they all run on an AI model underneath. The tool is the interface you interact with (chat window, IDE, terminal). The model is the AI brain doing the thinking. You can often choose which model to use inside the same tool.

Why it matters: Using a powerful model (like Opus) for a simple question wastes time and money. Using a fast model (like Sonnet) for a complex architecture task gives worse results. Match the model to the task complexity.
O
Claude Opus 4.6
The deep thinker — use for hard problems
Best for: complex architecture decisions, debugging tricky logic, multi-file refactoring, detailed code reviews, long-context analysis.

Think of it as your senior developer — slower but gives the most thoughtful, thorough answer. Use when quality matters more than speed.
S
Claude Sonnet 4.6
The everyday workhorse — use for most tasks
Best for: daily coding, component building, quick debugging, learning, writing docs, generating demos and artifacts.

Handles 80%+ of your work at near-Opus quality but much faster. This is your default — switch to Opus only when you need extra depth.
G
GPT-5.2 / 5.4
The fast builder — great for prototyping
Best for: quick prototyping, conversational brainstorming, terminal-heavy tasks (5.4), creative writing, web app generation.

GPT-5.4 is specifically strong for Codex (OpenAI's coding agent) and terminal execution. Good as an alternative perspective when Claude's answer doesn't fit.
G
Gemini 3.1 Pro
The value king — frontier quality, lower cost
Best for: large-context tasks (huge files/repos), research-heavy work, tasks within Google ecosystem, cost-sensitive high-volume work.

Matches Opus-level coding benchmarks at less than half the cost. Excellent if you work with Google Workspace tools.
?
Quick decision: which model do I pick?
"I need a quick answer or simple code" → Sonnet 4.6 (fast, good enough for most things)

"This is complex — architecture, tricky bug, or critical code" → Opus 4.6 (slower but much deeper reasoning)

"I'm processing a huge file or entire codebase" → Gemini 3.1 Pro (massive context window, great value)

"I want a second opinion or different approach" → GPT-5.2 / 5.4 (different strengths, fresh perspective)

"I'm not sure" → Start with Sonnet 4.6. If the answer isn't good enough, switch to Opus. That's what most developers do.
Models change fast — new versions come out every few months. The principle stays the same: use a fast model for simple tasks, a powerful model for hard ones. Don't overthink it — the right prompt matters more than the right model.
The most important part

4 rules that change everything

Tools don't matter if you use them wrong. These four rules make the difference between "AI is useless" and "AI saves me 2 hours a day."

1

Talk to it like a new team member

If a new developer joined your team today, you wouldn't say "fix the bug." You'd say "there's a bug in the dropdown component — it doesn't close when clicking outside. Here's the file." Same with AI.

VagueFix my CSS
ClearMy dropdown menu stays open when I click outside it. I'm using React with useRef and useEffect. Here's my current code: [paste]. What's wrong?
2

Have a conversation, not a one-shot

AI is like a rough sketch — the first answer is a starting point. The magic is in saying "now make it simpler" or "add error handling" or "that's not what I meant, I need X instead."

One-shotBuild complete login page → copy paste → done
IterativeBuild login form → add validation → make it RTL → match our tokens → add loading state
3

Give context upfront

AI doesn't know your project, your tech stack, or your team's conventions. The 30 seconds you spend giving context saves 10 minutes of fixing irrelevant answers.

No contextWrite a button component
With contextWrite a button for our Zoho Creator design system. SCSS with BEM, $zc- prefix tokens, RTL support, sm/md/lg sizes.
4

Trust but verify — always

AI is confident even when wrong. It's like a very fast junior developer — great speed, but you still need to review the code before it ships. Read it. Test it. Understand it.

Blind trustCopy AI code → paste into production → push
Smart trustRead the code → understand the logic → test locally → then use it
Copy & adapt these

6 prompt formulas that always work

Don't start from blank. Pick a formula, fill in your details, and you'll get 10x better results than "help me with this."

1
The context-first formula
▾

The single most important pattern. Start every prompt with context before your actual question. Think: "Who am I, what am I working on, what do I need?"

I'm working on [project] using [tech stack]. I currently have [what exists]. I need to [specific task]. Constraints: [any rules or preferences]
Real example
I'm working on a component library for Zoho Creator using SCSS with BEM naming. I have button and input components done. I need to create a dropdown that supports RTL, has sm/md/lg sizes, and follows the same pattern as the button. Keep transitions under 200ms.
2
The "act as" formula
▾

Give AI a role so it knows what depth and perspective to use.

Act as a [role] who specializes in [domain]. Review/Create/Analyze [the thing]. Focus on [specific aspects].
Real example
Act as a senior frontend developer who specializes in RTL layouts. Review this CSS file and list every property that needs RTL handling. Show the current code and the corrected version for each.
3
The "show me options" formula
▾

When you're not sure about the best approach, ask for 2-3 options with trade-offs. This is how you learn fastest.

I need to [task]. Show me [2-3] different approaches. For each: how it works, pros and cons, when to use it. My setup: [your constraints]
Real example
I need to handle theming in our SCSS design system. Show me 3 approaches — CSS custom properties, SCSS maps with mixins, and data-attribute selectors. For each, show a button component example and tell me which works best for RTL + multiple brand themes.
4
The "transform this" formula
▾

Give AI an input and tell it exactly what output format you want. Best for conversions and refactoring.

Here is [input: code, screenshot, file]. Transform into [desired output]. Rules: [list your requirements]
Real example
Here's a screenshot of a Creator PageBuilder form. Transform this into ZML markup. Use zc-form container, map fields to correct ZML input types, keep the two-column grid, add matching placeholder text.
5
The "review & fix" formula
▾

Paste your code and ask for a structured review. Great for learning — you'll see what's wrong and why.

Review this [code/component]: [paste code] Check for: [aspect 1, aspect 2, aspect 3] For each issue: show the line, explain why, show the fix.
Real example
Review this SCSS file. Check for: inconsistent token usage (should use $zc- variables), missing RTL support (should use logical properties), and hardcoded values. For each issue show the line, explain the problem, give the fix.
6
The "step by step" formula
▾

For complex tasks, ask AI to plan before doing. This prevents rushing to a wrong answer.

I need to [complex task]. Before writing code: 1. List the steps you'll take 2. Identify potential issues 3. Then implement step by step
Real example
I need to add RTL support to our dropdown component. Before code: list all CSS properties that need RTL handling, identify any JS logic that assumes LTR (like positioning), note icons that need mirroring. Then implement each fix, explaining each change.
Quick rules

Do's and don'ts

✓ Do this
✓ Give context — your project, tech stack, conventions
✓ Iterate — ask follow-ups, say "make it better"
✓ Paste your actual code, not a vague description
✓ Ask "why?" — learn from AI, don't just copy
✓ Read and test every line before using it
✓ Save prompts that worked — reuse them
✓ Break big tasks into smaller focused prompts
✗ Don't do this
✗ Don't copy AI code to production without reading it
✗ Don't use vague one-word prompts
✗ Don't share passwords, API keys, or secrets
✗ Don't assume the first answer is always right
✗ Don't dump 10 things in one prompt
✗ Don't give up after one bad answer — rephrase it
✗ Don't use AI as a crutch — use it as a multiplier
Learn from others

5 mistakes everyone makes at first

These are normal — just recognizing them puts you ahead.

1

Being too vague

"Fix my code" or "make a component" — AI doesn't know your project.

Fix: Say what you're building, what's broken, and what you expect. "The dropdown doesn't close on outside click. Here's my useEffect: [code]" gets you an answer in one try.
2

Accepting everything without reading

AI writes confident-looking code that might have subtle bugs or outdated patterns.

Fix: Read every line. If you don't understand something, ask "explain this part." Run it locally. Never push code you can't explain.
3

Using the wrong tool for the job

Using Claude Chat for tasks that need Claude Code, or Cursor for quick questions.

Fix: Quick question → Claude Chat. Debugging in browser → Browser Extension. Multi-file coding → Cursor. Repo-wide tasks → Claude Code. See the use case section below.
4

One giant prompt for everything

Putting 10 requirements in one message. AI tries to do everything and does nothing well.

Fix: One focus per message. First get the structure right, then styling, then interactions. Build up step by step.
5

Starting a new chat every time

Losing all the context AI built up about your project by opening a new conversation.

Fix: Keep one chat going for related tasks. AI remembers everything in the current conversation. Only start new for completely different topics.
